Getting started

From zero to running
in four steps.

No Docker required. No config files to write. No cloud account. Just install, register your project, and start.

01
Install the platform
One command installs three binaries to ~/bin/ — the daemon, the CLI, and the agent. Detects your OS and architecture automatically.
$ curl -fsSL https://harshmaury.github.io/Nexus/install.sh | bash
✓ engxd installed to ~/bin/engxd
✓ engx installed to ~/bin/engx
✓ engxa installed to ~/bin/engxa
✓ ~/bin added to PATH in ~/.bashrc
engxd is the background daemon. engx is the CLI you interact with. engxa connects remote machines to the same platform.
02
Register engxd as a system service
Makes the daemon start automatically at login and restart if it crashes. You never manage it manually again.
$ engx platform install
✓ engxd registered as system service
✓ engxd will start automatically at login
macOS uses launchd. Linux uses systemd. To skip this and start manually instead, run engxd &.
03
Register your project
engx init detects your language, finds the entry point, and generates a descriptor. Then engx register connects it to the platform. Supports Go, Node.js, Python, Rust, and .NET.
# Go to your project directory
$ cd ~/projects/my-api
 
# Detect language and generate descriptor
$ engx init
✓ .nexus.yaml written
name=my-api type=platform-daemon language=go
runtime: go run ./cmd/my-api/
 
# Register with the platform
$ engx register .
✓ Registered: my-api
✓ Service registered: my-api-daemon
You can edit .nexus.yaml after generation if the auto-detection isn't right. The file is plain YAML — set any command you want under runtime:.
04
Start and verify
engx run validates, starts, waits, and confirms in one command. If anything fails it tells you what/where/why and exactly what to type next.
$ engx run my-api
 
Project: my-api
Validating... ✓
Starting... ✓
Waiting... ···✓
 
Status: RUNNING
Services: 1/1 running
 
# Check the full platform
$ engx doctor
✓ engxd running uptime 0h 02m
✓ services 1/1 running
✓ db-integrity
✓ port-conflicts
If a service fails, engx run shows the failure reason and the exact command to fix it — no guessing required.
Interface

Four commands.
Ninety percent of your day.

Advanced commands are hidden until you need them. Start here.

01
engx run <project>
Validate, start, wait, and confirm in one command. Failures show what/where/why/next-step — not a stack trace.
Status: RUNNING
Services: 2/2 running
02
engx ps [project]
Project-level status. Shows outcomes, not internal IDs. Run without arguments to see everything at once.
✓ my-api running
✗ worker 1/2 — check: engx logs worker-daemon
03
engx logs <service>
Tail logs for any service. Every crash, restart, and state change is in the stream. Add --follow to watch live.
engx logs my-api-daemon
engx logs my-api-daemon --follow
04
engx doctor
Full platform diagnosis — daemon, running services, ports, permissions, database integrity, binary versions.
✓ engxd running uptime 4h 12m
✓ services 3/3 running

Platform

Built for systems
with moving parts.

⚡
Process control
Start, stop, restart with backoff. Services move to bounded maintenance on repeated failure — no infinite loops, no silent crashes.
🔍
Policy governance
Pre-execution gate blocks unsafe starts. Runtime findings surface failure patterns — high failure rate, unverified targets, stale projects.
📡
Structured events
Every state change is a typed event with trace ID, span ID, component, and outcome. Full history. Live SSE stream available.
🧠
AI insights
Sentinel correlates crash clusters, deploy timing, and dependency risk. Called only when you ask — never on background polls.
🔄
Automation engine
Trigger workflows on platform events or cron schedules. All executions logged with trace IDs — manual and automated unified.
🔒
Local-first
Everything runs on 127.0.0.1. No cloud dependency, no telemetry, no external calls during normal operation.
FAQ

Common questions.

What languages does engx support? ›
engx init auto-detects Go, Node.js, Python, Rust, and .NET. It finds the entry point and generates the correct run command automatically. For any other language, edit the generated .nexus.yaml and set the command manually — engx runs anything that can be started as a process.
Does it work with Docker or Kubernetes? ›
Yes — Docker and Kubernetes providers are supported. Set provider: docker or provider: k8s in your .nexus.yaml runtime section. The default process provider works without Docker and is the fastest path for local development.
What happens when a service crashes? ›
engx restarts it automatically with backoff — 5s, 15s, 30s between attempts. If it crashes repeatedly (default: 5 times), it enters a bounded maintenance state. Run engx services reset <id> to clear it and try again. Every recovery attempt is logged — nothing happens silently.
Can I run multiple projects at once? ›
Yes — that's the primary use case. Register each project with engx register <path>, then start them all with engx platform start. Use engx ps to see the status of every project at once. Each project can have multiple services managed independently.
Is my source code sent anywhere? ›
No. engx runs entirely on 127.0.0.1. The only external calls are checking GitHub Releases when you run engx upgrade, and the optional Sentinel AI feature (off by default). Your source code never leaves your machine.
What is the difference between engxd, engx, and engxa? ›
engxd is the daemon — runs in the background, manages services, tracks state, exposes the HTTP API. engx is the CLI — the only thing you interact with directly. engxa is the agent — connects a remote machine so you can manage its services from your local engx CLI.
